# CCTV Monitoring Research (Draft)

## 1. Цель исследования

Данное исследование проведено с целью сбора технических спецификаций и определения методов интеграции для разработки специализированной системы мониторинга **CCTV Monitoring Platform**.

Основной фокус направлен на оборудование **Hikvision** (около 90% парка), а также на обеспечение совместимости с **Dahua** и **Provision ISR**. Результаты исследования должны использоваться как основа для проектирования:

- слоя драйверов (**Driver Layer**)
- архитектуры опроса и событий
- модели данных статусов и метрик
- стратегии масштабирования

> Статус документа: **черновик / draft**.  
> Не использовать как окончательную API-спецификацию без лабораторной проверки на реальных устройствах.

---

## 2. Общая задача будущей системы

Проектируемая система **не является VMS (Video Management System)** и не предназначена для постоянного отображения, хранения или управления видеопотоками.

Назначение системы — **мониторинг состояния инфраструктуры видеонаблюдения**.

### Что должно мониториться

- **NVR/DVR**
  - доступность
  - аптайм
  - загрузка ресурсов (CPU / RAM)
  - общая работоспособность

- **Камеры**
  - статус online / offline
  - наличие связи с регистратором
  - состояние канала

- **Запись**
  - наличие или отсутствие процесса записи
  - состояние треков / каналов записи

- **Диски (HDD / SD)**
  - емкость
  - свободное место
  - состояние здоровья
  - ошибки хранения

- **Snapshot**
  - получение контрольного кадра
  - подтверждение, что камера действительно отдает изображение

- **События / алерты**
  - потеря видео
  - ошибки диска
  - саботаж
  - детекция движения
  - системные ошибки

## 3. Hikvision: способы интеграции

## 3.1 ISAPI (Intelligent Security API)

ISAPI рассматривается как основной проприетарный протокол Hikvision для интеграции по HTTP/HTTPS.

### Предполагаемые свойства

- архитектура REST-подобная
- транспорт HTTP / HTTPS
- аутентификация преимущественно **Digest Authentication**
- подходит для Polling-мониторинга

### Предварительно полезные endpoint'ы

> Ниже перечислены **предварительные** endpoint'ы, которые нужно подтвердить на реальном оборудовании.

- `/ISAPI/System/deviceInfo` — общая информация об устройстве
- `/ISAPI/System/status` — текущее состояние системы / нагрузка
- `/ISAPI/ContentMgmt/InputProxy/channels/status` — статус камер/каналов на NVR
- `/ISAPI/ContentMgmt/Storage/hdd` — информация о дисках
- `/ISAPI/Streaming/channels/<ID>/picture` — получение JPEG snapshot

### Ограничения

- документация может быть неполной в открытом доступе
- часть спецификаций предоставляется через Hikvision TPP / developer portal
- поведение endpoint'ов может отличаться между моделями и версиями прошивки

### Предварительный вывод

ISAPI выглядит лучшим кандидатом для **MVP-драйвера Hikvision**, но все конкретные endpoint'ы должны быть подтверждены:

1. официальной документацией для конкретной серии
2. живой проверкой на реальном NVR

---

## 3.2 Event API / Alarm Server

Механизм событий нужен для **быстрого получения инцидентов**, без постоянного частого опроса.

### Исследованные варианты

- **HTTP Event Stream / alertStream**
  - сервер мониторинга открывает постоянное соединение
  - устройство отправляет события по мере возникновения

- **Alarm Server / HTTP Listener / Push**
  - устройство отправляет уведомления на указанный URL
  - удобно для обработки событий с устройств

### Типы событий, которые ожидаются

- `videoloss`
- `diskerror`
- `tamper-detection`
- `motion detection / VMD`

### Предварительный вывод

На события **нельзя полагаться как на единственный источник правды**.  
Нужна гибридная схема:

- **event-driven** для быстрых тревог
- **polling / heartbeat** для подтверждения, что устройство не зависло “тихо”

## 3.3 SDK (HCNetSDK / другие варианты)

### HCNetSDK

Предварительно рассматривается как основной бинарный SDK Hikvision.

#### Плюсы

- максимальный доступ к функциям устройства
- богатые возможности интеграции
- пригоден для локальных и глубоких интеграций

#### Минусы

- тяжелее в разворачивании
- сложнее контейнеризировать
- бинарные зависимости
- выше стоимость сопровождения по сравнению с HTTP API

### Предварительный вывод

Для **облачной / серверной платформы мониторинга** разумнее начинать с **HTTP ISAPI**, а SDK использовать только там, где HTTP API не даст нужных данных.

## 4. ONVIF

ONVIF нужен как **межвендорный слой совместимости**.

### Что может быть полезно

- discovery
- базовая информация об устройстве
- snapshot URI
- RTSP URI
- базовые события / метаданные (в зависимости от профиля и реализации)

### Полезные профили

- **Profile S** — базовое видео
- **Profile G** — запись и работа с архивом
- **Profile T** — расширенное видео / кодеки / события
- **Profile M** — метаданные и аналитика

### Ограничения

- не гарантирует одинаковую глубину мониторинга между разными брендами
- детализация по HDD / storage health не является надежной кросс-вендорной гарантией
- производители реализуют ONVIF частично и по-разному

### Вывод

ONVIF — хороший **fallback / compatibility layer**, но не должен быть единственным способом мониторинга для Hikvision.

## 5. Другие производители

## 5.1 Dahua

### Предварительные выводы

- существует собственный HTTP / CGI API
- существует собственный SDK / NetSDK
- интеграция через API выглядит реалистичной

### Предварительно упомянутые точки

- `/cgi-bin/storageDevice.cgi?action=getDeviceAllInfo` — информация о дисках
- `/cgi-bin/snapshot.cgi?channel=<ID>` — snapshot

> Эти URI использовать только как гипотезы до подтверждения.

## 5.2 Provision ISR

### Предварительные выводы

- используется Ossia OS API / механизмы polling
- есть HTTP POST уведомления о событиях
- ONVIF поддерживается, глубина поддержки зависит от модели / прошивки

### Что требует проверки

- наличие стабильных endpoint'ов для мониторинга состояния HDD
- единообразие API между старыми и новыми моделями
- какие именно события можно принимать по push

## 7. Сравнение подходов к мониторингу

| Подход | Плюсы | Минусы | Масштабируемость |
|---|---|---|---|
| Polling | Надежность, предсказуемость | Задержка между опросами | Средняя / высокая при наличии воркеров |
| Event-driven | Мгновенные уведомления, меньше трафика | Потеря событий при сбое связи | Высокая |
| Heartbeat | Хорошо показывает доступность устройства | Не показывает все внутренние проблемы | Высокая |
| Snapshot Check | Позволяет проверить реальное изображение | Создает нагрузку на сеть и устройство | Низкая / средняя |
| RTSP / stream check | Проверяет доступность потока | Не всегда говорит о записи и здоровье устройства | Средняя |

### Вывод

Лучший вариант для платформы мониторинга — **гибридная архитектура**:

- heartbeat / polling как базовый механизм
- event-driven для быстрых тревог
- snapshot validation по расписанию и только там, где это действительно нужно

## 8. Архитектурные выводы

### Driver Layer обязателен

Ядро системы должно работать с унифицированными сущностями, например:

- `DeviceHealth`
- `CameraStatus`
- `DiskHealth`
- `RecordingStatus`
- `EventNotification`

Вендор-специфичная логика должна быть изолирована в драйверах.

### Hikvision как первый вендор для MVP

Hikvision логично брать первым, потому что:

- он занимает основную часть парка устройств
- по нему ожидается самая глубокая интеграция
- это даст максимальную практическую пользу уже на старте

### Рекомендуемая архитектура MVP

- `Device Registry`
- `Vendor Driver Layer`
- `Polling Scheduler`
- `Polling Workers`
- `Event Listener`
- `Metrics / State Storage`
- `Alerting`
- `Dashboard`

## 9. Риски и неизвестные зоны

- различия между прошивками и сериями устройств
- возможные отличия в endpoint'ах между моделями NVR
- ограниченная или нестабильная работа некоторых API без включения специальных настроек
- невозможность полностью полагаться на push-события на объектах за NAT / firewall
- неполная кросс-вендорность ONVIF для глубокого мониторинга состояния дисков и записи
